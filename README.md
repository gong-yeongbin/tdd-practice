# TDD란? (Test-Driven Development)

테스트 주도 개발(TDD, Test-Driven Development)은 **코드 작성에 앞서 테스트를 먼저 작성하는 개발 방법론**이다.  
일반적인 “기능 구현 → 테스트” 순서와 달리, “테스트 코드 작성 → 기능 구현 → 리팩터링” 단계를 반복하는 것이 핵심이다.

---

## TDD의 세 가지 단계

1. **Red 단계**
   - 아직 구현되지 않은 기능에 대해 실패하는 테스트를 먼저 작성한다.
   - 이 시점의 테스트는 당연히 실패(Red) 상태여야 한다.

2. **Green 단계**
   - 테스트를 통과할 수 있는 최소한의 코드를 작성한다.
   - 설계나 구조보다는 “일단 테스트를 통과시키는 것”에 집중한다.

3. **Refactor 단계**
   - 중복을 제거하고 가독성을 높이며 코드를 리팩터링한다.
   - 리팩터링 후에도 기존 테스트가 모두 통과해야 한다.

---

## TDD의 장점

1. **코드 품질 향상**
   - 코드는 항상 테스트를 통과해야 하므로 더 견고해진다.
   - 테스트가 기능의 명세서 역할을 하여 코드의 의도를 명확히 이해할 수 있다.

2. **디버깅 시간 절약**
   - 테스트가 실패하는 시점을 기준으로 문제를 빠르게 좁혀갈 수 있다.
   - 변경 후 바로 테스트를 돌려 보면서 디버깅 시간을 줄일 수 있다.

3. **리팩터링이 쉬움**
   - 테스트가 안전망 역할을 하므로 마음 놓고 코드를 리팩터링할 수 있다.
   - 변경 후에도 테스트가 통과하는지로 기능이 깨졌는지 쉽게 확인 가능하다.

4. **빠른 피드백 루프**
   - 코드를 작성할 때마다 즉시 테스트로 검증해볼 수 있다.
   - 기능이 의도한 대로 동작하는지 빠른 피드백을 받을 수 있다.

---

## TDD의 단점

1. **초기 학습 곡선**
   - “테스트를 먼저 작성한다”는 방식이 익숙하지 않을 수 있다.
   - 좋은 테스트를 설계하는 법을 익히는 데 시간이 필요하다.

2. **개발 속도 저하(초기)**
   - 테스트와 기능 구현을 번갈아 작성해야 해서 초반 개발 속도가 느려질 수 있다.
   - 다만 장기적으로는 버그 감소와 유지보수 비용 절감으로 상쇄되는 경우가 많다.

3. **모든 상황을 다 다루기 어려움**
   - UI, 복잡한 비즈니스 로직, 외부 시스템 연동 등은 테스트 작성이 까다로울 수 있다.
   - 이 때문에 TDD가 항상 전 영역에 100% 적용되기는 어렵다.
